Quick Troubleshooting — ERR_EMPTY_RESPONSE & ERR_ADDRESS_UNREACHABLE

Quick Troubleshooting

Error found

Two reported errors: ERR_EMPTY_RESPONSE (5.9.8.9) and ERR_ADDRESS_UNREACHABLE (1.2.3.4)

1) ERR_EMPTY_RESPONSE

Message you saw
This page isn’t working
http://5.9.8.9/ didn’t send any data.
Browser error: ERR_EMPTY_RESPONSE

What it means
The browser opened a TCP connection (or attempted to) but the server closed the connection without sending any HTTP response bytes (no headers, no body). The browser reports "empty response".

Common causes

  • Server process crashed or closed socket immediately after accept.
  • Firewall, proxy, or middlebox dropped or blocked the response.
  • Server listening on the port but not speaking HTTP (raw socket behavior).
  • Network device (NAT, router) interfering with the connection.

Quick checks you can run

  1. Open a terminal/command prompt on your machine.
  2. Try a simple TCP check with curl (shows low-level details):
    curl -v http://5.9.8.9/
  3. Try ping to see if host responds to ICMP:
    ping 5.9.8.9
  4. Run traceroute to see where packets stop:
    traceroute 5.9.8.9 (macOS/Linux) or tracert 5.9.8.9 (Windows)
  5. Try connecting with telnet to the HTTP port to observe raw behavior:
    telnet 5.9.8.9 80 — if it connects and immediately closes, server sent nothing.

How to reproduce (developer / tester)

  1. Run a tiny server that accepts TCP and closes without writing any HTTP response (Node/Python examples below).
  2. Point your browser to that server's IP:port and observe ERR_EMPTY_RESPONSE.

Simple Node example (local reproduction)
Run this on your machine (Node.js installed):

const net = require('net');
const server = net.createServer((socket) => {
  // accept then immediately destroy the socket (no HTTP response)
  socket.destroy();
});
server.listen(8080, '0.0.0.0', () => {
  console.log('Listening on 0.0.0.0:8080 (will accept then close)');
});
Then open http://127.0.0.1:8080/ in your browser to see an empty response behavior.

Troubleshooting / fixes

  • If you control the server: check server logs, ensure the HTTP server process is running and not crashing, verify it writes proper HTTP headers and body.
  • Check reverse proxies (nginx, haproxy) and their logs — misconfiguration can drop responses.
  • Temporarily disable firewall/antivirus or test from another network to rule out local filtering.
  • Test with curl -v to capture handshake and any immediate socket close details.

2) ERR_ADDRESS_UNREACHABLE

Message you saw
This site can’t be reached
http://1.2.3.4/ is unreachable.
Browser error: ERR_ADDRESS_UNREACHABLE

What it means
The browser (or OS network stack) determined the destination address cannot be reached from your network. This is different from DNS failure — the IP itself is unreachable (no route, no ARP, or blocked by network devices).

Common causes

  • Routing problem: no route to that IP from your ISP or local network.
  • Network-level blocking by ISP, corporate firewall, or router ACLs.
  • IP is reserved or not assigned on the public internet (or filtered by upstream provider).
  • Local misconfiguration (wrong gateway, VPN/proxy interfering).

Quick checks you can run

  1. Ping the IP:
    ping 1.2.3.4
  2. Traceroute to see where packets stop:
    traceroute 1.2.3.4 or tracert 1.2.3.4
  3. Check your local route table:
    ip route (Linux/macOS) or route print (Windows)
  4. Try from another network (mobile hotspot) to confirm whether the problem is local to your network or global.

How to reproduce

  1. Block outbound traffic to 1.2.3.4 on your router or firewall and attempt to open the URL — browser will show unreachable.
  2. Use a network with no route to that IP (some lab networks) and attempt to connect.

Troubleshooting / fixes

  • Restart your router and modem; disable VPN/proxy temporarily.
  • Check local firewall/antivirus rules that might block the destination IP.
  • Run traceroute; if packets stop at your ISP, contact your ISP and provide the traceroute output.
  • If you control the destination IP, verify the host is online, has correct routing, and responds to ARP/ICMP/TCP.

Useful commands summary

# Check raw HTTP behavior
curl -v http://5.9.8.9/

# Ping the host
ping 5.9.8.9
ping 1.2.3.4

# Traceroute to see where packets stop
traceroute 5.9.8.9   (macOS / Linux)
tracert 5.9.8.9      (Windows)

# Raw TCP connect
telnet 5.9.8.9 80

# Check local routes
ip route             (Linux / macOS)
route print          (Windows)
